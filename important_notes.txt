BIOS:
What is a memory?
A: Memory is the piece of hardware used by computers to store information.

What is a bootloader?
A: Bootloader is a small computer program that loads the kernel of an operating system.

What is BIOS?
A:
1) Stands for Basic Input Output system.
2) It is a kernel in itself, it initializes hardware.
3) It looks for bootloader to boot by searching all storage mediums for the 0x55AA boot signature.
4) It is 16 bit code. Only 16 bit code can execute it properly.

Booting execution order: BIOS -> Bootloader -> kernel

Real Mode:
- Is a compatibility mode that all modern intel processors started when they are switched on.
- Can only excess 1 MB of RAM.
- Based on the original x86 design.
- No security.
- 16 bits accessible at one time.
- Memory access is done through segmentation memory model.
- We can only access 8 bit and 16 bit registers.

Segmentation Memory Model:
- Memory is addressed by segment and offset.
- Programs can be loaded in different areas of memory but run without problems.
- Multiple segments are available through the use of segment registers.

8086 Segment Registers:
- CS: Code Segment
- DS: Data Segment
- ES: Extra Segment
- SS: Stack Segment

Calculating Absolute Offset:
- Take the segment register multiply it by 16 and add the offset.
- Stack goes from high memory to low memory and heap goes from low memory to high memory. Stack grows downwards and heap grows upwards in x86.

Interrupts:
- Interrupts are like subroutines, don't need to know its memory address to invoke them.
- Interrupts are called through the interrupt numbers.

What happens when you invoke an interrupt?
- Processor is interrupted.
- Old state saved on stack.
- Interrupt executed.

Interrupt vector table explained
- Table describing 256 interrupt handlers.
- Entries contain 4 bytes (offset:segment)
- Interrupts are in numerical order in the table.

Interrupt Vector table:
- It is organized like this:
Offset (2 bytes) | Segment (2 bytes) | Offset | Segment | Offset | Segment | ... [256 times]
- Each interrupt is 4 bytes.
- Example: Interrupt 0 is at address 0x00
- Interrupt 1 at address 0x04
- Interrupt 2 at address 0x08
- Interrupt 0x13 at offset 0x13 * 0x04 = 76 (Decimal), processor looks at offset 76 in RAM:
  76-77 = Offset
  78-79 = Segment

Filesystem:
- Filesystems are kernel implemented they are not the responsibilty of the hard disk.
- Implementing a filesystem requires the kernel programmer to create a filesystem driver for the target filesystem.
- Data is read and written in sectors typically 512 byte blocks.
- Reading the sector of a disk will return 512 bytes of data for the chosen sector.
- Earlier sectors are read and written by specifying a "head", "track" and "sector".
- Currently, we use LBA (Logical block address) as a modern way of reading from a hard disk rather than specifying "head", "track" and "sector", we just specify a number that starts from zero.
- LBA allows us to read from disk as if we are reading blocks from a very large file.
- LBA 0 = first sector on the disk
- LBA 1 = second sector on the disk

Calculating LBA:
- Reading byte 58376: LBA = 58376/512 = 114
- Offset = 58376 % 512 = 8
- 114 * 512 = 58368
- 58368 + 8 = 58376

Protected Mode:
- Is a processor state in x86 architectures which gives access to memory protection, 4GB address space.
- Can provide memory and hardware protection.
- Different memory schemes.
- 4 GB of memory directly addressable.

Memory and Hardware Protection:
- Protected Mode allows you to protect memory from being accessed.
- Can prevent user programs talking with hardware.
- Ring 0: Kernel (Most privileged), Ring 1 and Ring 2 (Device drivers) (lesser privileged).
- Ring 3: Applications (Least privileged).

Selector Memory Scheme:
- Segmentation registers become selector registers.
- Selectors point to data structures that describe memory ranges and the permissions (ring level) 

Paging Memory Scheme
- Memory is virtual and what you address can point to somewhere entirely different in memory.
- Memory protection is easier to control.

4GB of Addressable Memory
- We gain access to 32-bit instructions and can easily work with 32-bit registers
- We can address up to 4GB of memory and no longer limited to 1 MB of memory provided by real mode.

Text mode:
- Allows you to write ascii to video memory. Ascii can be written into memory starting at address 0xB800 for coloured displays.
- Supports 16 unique colors. Each ascii character written to this memory has its pixel equivalent outputted to the monitor.
- No need to set individual screen pixels for printing characters.
- Each character takes up 2 bytes:
  1) Byte 0: Ascii character
  2) Byte 1: Colour code

Interrupt Descriptor Table:
- Describes how interrupts are invoked in protected mode.
- Can be mapped anywhere in memory.
- Different from interrupt vector table.
- Contains the following things:
  1) Offset (Bit 46-63) - Higher part pf the offset to execute
  2) P (Bit 47) - Should be set to xero for unused interrupts
  3) DPL (Bit 45-46, Descriptor Privilege Level) - The ring level the processor requires to call this interrupt   
  4) S (Bit 44, Storage Segment) - Should be set to zero for trap gates  
  5) Type (Bit 40-43, Gate Type) - The type of gate this interrupt is treated as  
  6) 0 (Bit 32-39, Ununsed 0-7) - Ununsed bits in this structure
  7) Selector (Bit 16-31, Selector 0-15) - The selector this interrupt is bounded to, the kernel code selector. 
  8) Offset (Bit 0-15, Offset 0-15) - The lower part of the offset to execute.

Print function semantics:
- void print -> terminal_writechar -> terminal_putchar -> terminal_makechar

Programmable interrupt controller (PIC):
- Allows hardware to interrupt the processor state.
- Programmable
- Requires interrupt acknowledgement.
- Allows different types of hardware to interrupt the processor such as hard disk, keyboard and more.
- Standard ISA (Instruction set architecture) IRQ's (Interrupt request):
  IRQ | Description
  0   | Timer Interrupt
  1   | Keyboard Interrupt
  2   | Cascade (used internally by the two PICs. never raised)
  3   | COM2 (if enabled)
  4   | COM1 (if enabled)
  5   | LPT2 (if enabled)
  6   | Floppy disk
  7   | LPT1
  8   | CMOS real time clock (if enabled)
  9   | Free for peripherals / legacy SCSI / NIC
  10  | Free for peripherals / SCSI / NIC 
  11  | Free for peripherals / SCSI / NIC 
  12  | PS2 Mouse
  13  | FPU / Coprocessor / Inter-processor
  14  | Primary ATA Hard Disk
  15  | Secondary ATA Hard Disk
  
Understanding IRQ'S:
- IRQ's are mapped to a starting interrupt for example 0x20.
- IRQ 0 would be interrupt 0x20
- IRQ 1 would be interrupt 0x21
- IRQ 2 would be interrupt 0x22

Master vs Slave:
- The system has 2 PIC (programmable Interrupt Controller) one for master ports and other for slave ports.
- Master handles IRQ 0-7.
- Slave handles IRQ 8-15.

PIC Control Ports:
- 0x20 and 0x21: Master IRQ'S
- 0xA0 and 0xA1: Slave IRQ's

Interrupt Acknowledgement:
- You must let PIC controller know when you handled the interrupt.

Heap memory:
- Is a giant memory region that can be shared in a controlled manner.
- You can ask the heap for memory and tell the heap when your done with that memory.
- Heap implementations are essentially system memory managers.
- In C programming language, you can point to any memory address in RAM regardless if you can access it or not.

Malloc:
- Returns a memory address of memory that we can write too (it becomes ours).
- Ensures that any other time your program calls "malloc", it does not return a memory address that is unavailable.

Free:
- Accepts the memory address that we want to free.
- Marks that memory address and all associated blocks as free.
- Next time someone calls "malloc", we can safely end up with a previous address that was used.

Memory limits for a 32 bit kernel:
- In protected mode, we have certain restrictions, the processor is in 32 bit state.
- ONly have access to 32 bit memory addresses allowing us to address to a max of 4.29 GB of RAM.

Memory of an uninitialized system:
- Video memory takes up portions of RAM.
- Hardware memory takes up portions of RAM.
- Unused parts of RAM are available to use.
- An array of uninitialized memory is available to us from address "0x01000000"

Simplest possible heap implementation:
- Start with a start address and call it a "current address" point it somewhere free.
- Any call to malloc gets the current address stores it in a temporary variable called "tmp".
- Now the current address is incremented by the size provided to "malloc".
- Temporary variable called "tmp" that contains allocated address is returned.
- Current_address now contains the next address for "malloc" to return when "malloc" is called again.
- Benefits: Easy to implement but memory can never be released which can lead to system being unusable and requiring a reset.

Our heap implementation:
- Will consist of a giant table which describes a giant piece of free memory. This table will describe which memory is taken, which memory is free and so on. We will call this the "entry table".
- Will have another pointer to a giant piece of free memory, this will be the actual heap data itself that users of "malloc" can use. We will call this the "data pool". If our heap can allocate 100 MB of ram then the heap data pool wil be 100MB in size.
- Our heap implementation will be block based, each address returned from "malloc" will be aligned to 4096 and will at least be 4096 in size.
- If you request to have "50" bytes 4096 bytes of memory will be returned.

The entry table:
- Composes of an array of 1 byte values that represent an entry in our heap data pool.
- Array size is calculated by taking the heap data pool size and dividing it by our block size of 4096 bytes. We are left with the total number of entries we need in our array.
- We want a 100MB heap then the math is: 100MB/4096 = 25600 bytes in our entry table.
- If our heap data pool is at address 0x01000000 then entry zero in our table will represent address 0x01000000.
- Entry one will represent 0x01001000, entry two will represent 0x01002000 where 0x1000 represents 4096 bytes.

The entry structure:
- Bit structure of 8 bit entry in our table
(Upper 4 bits are flags)  (Lower 4 bits are the entry type)      
(7) HAS_N | IS_FIRST | 0 | 0 | ET_3 | ET_2 | ET_1 | ET_0 (0)

HAS_N = Set if the entry to the right of us is part of our allocation.
IS_FIRST = Set if this is the first entry of our allocation.

Each entry byte describes 4096 bytes of data in the heap data pool.

Entry types:
- HEAP_BLOCK_TABLE_ENTRY_TAKEN: The entry is taken and the address cannot be used.
- HEAP_BLOCK_TABLE_ENTRY_FREE: The entry is free and may be used.

The data pool:
- Simply a raw flat array of thousands or millions of bytes that our heap implementation can give to people who need memory.

Malloc example:
- Firstly we assume we set our heap data pool to point to address 0x01000000.
- We assume our heap is 100MB in size.
- We assume we have 25600 entries in our entry table that describe our 100 MB of data in the data pool. 100 MB/4096 = 25600

Malloc example - Memory allocation process
- Take the size from malloc and calculate how many blocks we need to allocate for this size. If the user asks for "5000" bytes we will need to allocated 8192 bytes because our implementation works on 4096 byte blocks. 8192 bytes is 2 blocks.
- Check the entry table for the first entry we can find that has a type of HEAP_BLOCK_TABLE_ENTRY_FREE, meaning that the 4096 block that this entry represents is free for use.
- Since we require 2 blocks we also need to ensure the next entry is also free for use otherwise we will need to discard the first block we found and look further in our table until we find at least 2 free blocks that are next to each other.
- Once we have 2 blocks we mark those blocks as taken.
- We now return the absolute address that the starting block represents. Calculation: (heap_data_pool_start_address + (block_number * block_size))

a) Step 1 - Finding the total blocks
- Get the size provided to malloc. For example value "5000" we align it to "4096" we get the value 8192.
- Now we just divide 8192 by our block size of "4096" which gives us 8192/4096 = 2. Two blocks to allocate.

b) Step 2 - Find 2 free blocks in the table
- 0xC1 = Block taken, first block, has more blocks for this allocation.
- 0x41 = Block taken, first block, no more blocks for this allocation.
- 0x81 = Block taken we have more blocks for this allocation, we are not the first block for this allocation.
- 0x01 = Block taken, we are not the first block for this allocation, no more blocks for this allocation.
- 0x00 = Block free or (lower 4 bits = 0) block free.

c) Step 3 - Calculate the absolute address the programmer can use
- Next step is to return the absolute address in our data pool. Entries in our table start from zero just like arrays.
- 22 * block_size(4096) = 90122 decimal. This is our offset.
- Our heap data pool starts at address 0x01000000 so we just add the offset 0x01000000 + 90122 decimal = 0x101600A.
- Our absolute address our "malloc" implementation has found for us is 0x101600A

Free example:
- Calculate the block number based on the address provided to us to free.
- Go through the entry table starting at the block number we have calculated, set each entry to "0x00" until we reach the last block of the allocation.
- We know how many blocks we need to free because the current block we are freeing will not have the "HAS_N" bit set in the entry byte.

Diadvantages of our implementation:
- We allocate in memory blocks meaning misaligned sizes requested from our heap will result in wasted lost bytes.
- Memory fragmentation is possible.

Paging:
- Allows us to remap memory addresses to point to other memory addresses.
- Can be used to provide the illusion we have the maximum amount of RAM installed.
- Can be used to hide memory from other processes.

Remapping memory:
- Paging allows us to remap one memory address to another, so 0x100000 could point to 0x200000.
- Paging works in 4096 byte block sizes by default. The blocks are called pages.
- When paging is enabled the MMU (Memory Management Unit) will look at your allocated page tables to resolve
virtual addresses into physical addresses.
- Paging allows us to pretend memory exists when it does not.

Virtual addresses vs Physical addresses:
- Are addresses that are not pointing to the address in memory that their value says they are. Virtual address 0x100000 might point to physical address 0x200000.
- Physical addresses are absolute addresses in memory whose value points to the same address in memory.
- Essentially virtual address and physical address are just terms we used to explain how a piece of memory is being accessed.

Structure Overview:
- 1024 page directories that point to 1024 page tables.
- 1024 page table entries per page table.
- Each page table entry covers 4096 bytes of memory.
- Each "4096" byte block of memory is called a page.
- 1024 * 1024 * 4096 = 4GB of addressable memory.

Page Directory Structure:
- Holds a pointer to a page table.
- Holds attributes.
- P: The bit is set if the page exists in real memory. If this page is not actually available, then this bit should be set to zero. If someone accesses this memory, a "Page fault" will occur.
- R: If this bit is set the page is readable and writable if its not set then this page is only readable.
- U (User Supervisor Bit): If set then this page can be accessed by all privilege ring levels. If not then only supervisor ring levels can access this page.
- W: If set, then write through caching is enabled, if not, then write back is enabled instead.
- D: Cache disable bit. Set to 1 to disable page caching.
- A: Set to 1 by the CPU if this page is accessed.
- S: Set to 0 for 4KB pages, set to 1 for 4 MB pages.
- Page table 4-KB Aligned Address: Physical 4096 byte aligned address to the start of a page table.

Page Fault Exception:
- The CPU will call page fault interrupt 0x14 where there was a problem with paging.
The exception is invoked:
- If you access a page in memory that does not have its "P (Present)" bit set.
- Invoked if you access a page that is for supervisor but you are not supervisor.
- Invoked if you write to a page that is read only and you are not supervisor.

Hiding memory from processes:
- If we give each process its own page directoy table then we can map the memory for the process however we want it to be. We can make it so the process can only see itself.
- Hiding memory can be achieved by switching the page directories when moving between processes.
- All processes can access the same virtual memory addresses but they will point to different physical addresses.

Illusion of more memory:
- We can pretend we have the maximum amount of memory even if we do not.
- This is achieved by creating page tables that are not present. Once a process accesses this non-present address a page fault will occur. We can then load the page back into memory and the process had no idea.
- 100 MB system can act as if it has access to full 4GB on a 32 bit architecture.

Benefits to paging:
- Each process can access the same virtual memory addresses, never writing over each other.
- Security is an added benefit as we can map out physical memory that we don't processes to see.
- Can be used to prevent overwriting of sensitive data such as program code.
